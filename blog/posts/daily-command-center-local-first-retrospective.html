Making a Day Recoverable: A Daily Command Center Retrospective

I initially wanted to see what mattered before leaving home. The real engineering question became how tasks, time and decisions could remain trustworthy after reloads, restarts, migrations and mistakes.

23.07.2026Engineering8 Min.

Einen Tag wiederherstellbar machen: Rückblick auf das Daily Command Center

Am Anfang wollte ich nur vor dem Weggehen erkennen, was wirklich wichtig war. Die eigentliche Entwicklungsfrage lautete bald: Wie bleiben Aufgaben, Zeit und Entscheidungen nach Neuladen, Neustart, Migration und Fehlern vertrauenswürdig?

这次开发最有意思的地方，是“个人效率工具”最终没有被做成一个更漂亮的待办清单。它被做成了一套小型的、本地优先的工作状态系统：早晨承诺，白天只专注一个任务，结束时留下证据，晚上关闭这一天。

10个功能与修复提交
35项自动化测试
7,580新增代码与测试行
14个旧任务安全迁移
Schema 7最终数据库版本

一、框架性的优势：把任务列表变成状态机

Check-in容量、最多三个结果、所有遗留任务的去向
→
Focus唯一进行中任务、唯一计时器、明确的下一步
→
Checkout实际产出、剩余估时、偏差原因和下次动作
→
Daily Close关闭开放会话，生成不可变日快照

这种框架的第一项优势是减少上下文切换。页面不问“你拥有哪些任务”，而是问“你现在处于哪一个工作状态”。最多三个 Daily Outcomes 不是为了限制野心，而是让容量成为真实约束。

第二项优势是所有数字都可解释。Initial Estimate 不会被后续修改覆盖；暂停、恢复、Checkout 和 Time Adjustment 都成为追加事实。现在显示的用时不是一个孤立数字，而可以沿历史回答“它为什么是这个数字”。

第三项优势是未来可迁移。UI 不直接操纵表，而是调用带 UUID、幂等键、版本和 Actor Context 的语义命令。今天后端是本机 SQLite，未来转为在线数据库和登录用户时，不必推翻任务语言。

第四项优势是本地优先的安全边界。正式服务只监听 127.0.0.1；外部网页不能随意向 loopback 写任务；来源刷新不会在页面请求里读取本地平台。便利性被刻意让位给可控性。

二、成本：真正昂贵的是可靠性，不是页面

可观测的核心实现窗口：从第一个运行时提交到最终迁移修复约 75 分钟；包含后续可读性调整约 86 分钟。这个数字不包括此前的需求澄清、原型讨论和规格设计，因此不能当作完整人时。

可量化的工程量：10 个提交、35 项测试、22 个变更文件、约 7,580 行新增代码与测试。实际模型/API 金额没有可靠账单数据，所以本文不虚构货币成本。

页面本身并不昂贵。昂贵的是：第二个浏览器标签页不能启动第二个计时器；旧数据库升级失败不能破坏原库；旧版开放计时必须能恢复；跨午夜片段只能按当天重叠部分计入快照；重复点击不能制造第二条事实。

安全和迁移代码让系统显著变大。对于一个人使用的工具，这很容易被批评为过度工程。但如果它长期承担时间分析，错误数据会持续训练出错误判断。此时“简单但会悄悄错”比“稍复杂但可解释”更贵。

三、好处与不好处

得到的好处付出的代价
刷新、浏览器和服务重启后可恢复需要事件、版本、会话片段和恢复命令
预计与实际差异可长期分析不能直接覆盖错误数据，只能追加修正
旧任务得到保留并进入新流程迁移逻辑成为必须长期维护的产品功能
本地数据不向互联网暴露手机暂时不能直接通过局域网访问
未来上线时已有稳定领域接口初版代码量大于普通 CRUD 待办应用
一天可以明确关闭，不把工作带到夜里Daily Close 要求用户诚实填写证据和下一步

还有一项明确的未完成：Gmail、日历、iCloud Notes、Voice Collector、Books 和浏览历史的可信自动同步仍属于后续阶段。初版完成的是手动任务与每日工作闭环，不应把“接口已经规划”描述成“平台已经实时接通”。

四、开发经历：测试全绿以后，问题才真正出现

第一个可用版本有 28 项测试，而且真实 Chrome 在 390 × 844 下没有横向溢出。看起来已经可以安装。但两条并行审查发现：旧任务系统和新语义账本仍能分别启动计时，因此“只能有一个计时器”的约束并不是全局的；Daily Snapshot 还可能重复计算跨日任务的历史时间。

修复后，复审又发现更窄的升级路径：旧开放计时被迁成不确定片段后，如果只关闭新片段而不退休旧表记录，所有新计时仍会被永久阻止。再往后一层，是已经运行过 schema 6 的用户可能跳过新增迁移。最终数据库升级到 schema 7，并为这些历史状态增加了精确测试。

这次最重要的经验：一个 invariant 必须跨越所有旧接口、迁移状态和恢复路径成立。只在新表里加唯一索引，并不能证明整个产品只有一个计时器。

最后，所有工程门禁通过后，我仍然说了一句：“字太小，我看不清。”这不是小问题，而是最诚实的验收。系统通过了计时、恢复、迁移和手机宽度测试，却没有一开始覆盖大量 9–12px 的辅助文字。最终正文和任务提高到 18px，小标签提高到 15px，按钮提高到 17px/48px。

这提醒我：自动化测试负责证明系统没有背叛规则，人的身体负责证明产品是否真的能用。二者缺一不可。

五、如果重新做一次，我会保留什么、减少什么？

一定保留：先定义日循环

Check-in、Focus、Checkout、Daily Close 是产品骨架。没有骨架，接入再多平台也只是制造更大的收件箱。

一定保留：追加式历史

估时学习依赖原始事实。覆盖旧数字会让长期分析失去意义。

更早处理：旧数据迁移

迁移不是上线前的清理工作，而是从第一张领域图开始就必须进入验收的用户路径。

更早处理：人体可读性

除像素门槛外，应在第一轮原型就以真实距离、真实光线和疲劳状态阅读所有辅助文字。

减少：过早连接所有来源

邮件、日历和 Notes 只有在核心状态机稳定后才值得接入，否则自动化只会加速噪音进入系统。

减少：用“测试通过”代替完成

测试、双轨审查、真实服务、迁移后数据对账和人的视觉体验应共同构成交付证据。

结论：个人软件也需要一条可信的责任链

Daily Command Center 的价值不在于它比 Reminders 多几个按钮，而在于它试图建立一条责任链：早晨我承诺什么，现在我做什么，时间去了哪里，结束时留下了什么，明天从哪里继续。

框架给了这条责任链稳定的形状；事件账本让它可以解释；本地优先让它保持私密；固定点审查阻止“看起来完成”成为真正交付。代价是更多代码、更慢的第一版和更严格的使用纪律。但如果目标是长期认识自己的时间，而不是短暂获得掌控感，这个代价是值得的。

关键词：状态机 = state machine / Zustandsautomat；追加式账本 = append-only ledger / unveränderliches Ereignisprotokoll；幂等 = idempotency / Idempotenz；恢复点 = recovery point / Wiederherstellungspunkt；固定点审查 = fixed-point review / Fixpunktprüfung。

I did not want another attractive to-do list. I wanted a system that could answer four questions: What did I commit to? What am I doing now? Where did the time go? Where should I continue tomorrow?

The framework

The product is organized as a state machine: Check-in limits the day to three outcomes; Focus permits one in-progress task and one running timer; Checkout records output, remaining estimate and the next action; Daily Close produces an immutable snapshot. An append-only ledger preserves starts, pauses, resumptions, checkouts and corrections instead of overwriting history.

Semantic commands use stable IDs, idempotency keys and optimistic versions. SQLite is the local authority today, but the domain boundary can later support authentication and an online database without redefining what “start”, “pause” or “checkout” means.

The observable cost

The core implementation commits span about 75 minutes; including the later readability correction, about 86 minutes. This excludes earlier discovery, prototypes and specification work, so it is not total labour. The result involved 10 commits, 35 automated tests, 22 changed files and roughly 7,580 added lines of code and tests. No reliable model/API billing record was available, so no monetary cost is invented.

The expensive work was not the interface. It was enforcing one timer across old and new systems, preserving a database during failed migration, recovering an uncertain legacy segment, splitting work across midnight, and making retries create one fact rather than two.

Benefits and disadvantages

The system survives reloads and service restarts, explains estimate gaps, preserves old tasks and keeps personal data on the Mac. In exchange, it has more code than a CRUD task app, requires disciplined checkout, temporarily gives up unauthenticated phone access and must maintain migration logic as a permanent product feature.

Automatic trusted synchronization with mail, calendars, Notes, Voice Collector, Books and browser history is not finished. The initial version completes the manual daily loop; it does not pretend that planned connectors are already live.

What the development taught me

Twenty-eight green tests did not reveal that old and new ledgers could each run a timer. Review did. Later review found a schema-6 upgrade edge that could keep an old timer open forever. The final version reached schema 7 and 35 tests.

Then the human acceptance test found something simpler: much of the secondary text was too small. The system passed recovery and mobile-width gates, yet it was uncomfortable to read. Body and task text moved to 18px, labels to 15px and controls to 17px with 48px targets.

Automation proves that software respects its rules. A human body proves that the product is actually usable.

Conclusion

The framework's value is a trustworthy chain of responsibility across a day. It costs more code, a slower first release and stricter habits. For long-term learning about one's own time, that cost is more honest than a simpler tool that silently corrupts its history.

Ich wollte keine weitere schöne Aufgabenliste. Das System sollte vier Fragen beantworten: Wozu habe ich mich verpflichtet? Woran arbeite ich jetzt? Wohin ging die Zeit? Wo mache ich morgen weiter?

Das Framework

Das Produkt ist als Zustandsautomat aufgebaut: Check-in begrenzt den Tag auf drei Ergebnisse; Focus erlaubt genau eine laufende Aufgabe und einen Timer; Checkout dokumentiert Ergebnis, Restschätzung und nächsten Schritt; Daily Close erzeugt einen unveränderlichen Tages-Snapshot. Ein append-only Ereignisprotokoll bewahrt Start, Pause, Fortsetzung, Checkout und Korrektur.

Semantische Befehle verwenden stabile IDs, Idempotenzschlüssel und optimistische Versionen. Heute ist SQLite die lokale Autorität. Später kann dieselbe Domänengrenze Anmeldung und Online-Datenbank tragen, ohne „Start“, „Pause“ oder „Checkout“ neu zu definieren.

Der messbare Aufwand

Zwischen dem ersten Laufzeit-Commit und der letzten Migrationskorrektur lagen ungefähr 75 Minuten; mit der späteren Lesbarkeitskorrektur etwa 86 Minuten. Frühere Klärung, Prototypen und Spezifikation sind darin nicht enthalten. Das Ergebnis umfasst 10 Commits, 35 automatisierte Tests, 22 geänderte Dateien und ungefähr 7.580 neue Code- und Testzeilen. Für Modell- oder API-Kosten lag keine verlässliche Abrechnung vor.

Teuer war nicht die Oberfläche. Teuer waren ein global eindeutiger Timer, sichere Migration, Wiederherstellung unsicherer Alt-Segmente, Zeitaufteilung über Mitternacht und idempotente Wiederholungen.

Vorteile und Nachteile

Das System übersteht Browser- und Dienstneustarts, erklärt Schätzabweichungen, übernimmt alte Aufgaben und hält persönliche Daten auf dem Mac. Dafür ist es größer als eine CRUD-App, verlangt konsequentes Checkout, verzichtet vorerst auf ungeschützten Smartphone-Zugriff und muss Migration dauerhaft als Produktfunktion pflegen.

Die vertrauenswürdige automatische Synchronisierung mit E-Mail, Kalender, Notes, Voice Collector, Books und Browser-Historie ist noch nicht abgeschlossen. Die erste Version liefert den manuellen Tageskreislauf und behauptet nicht, dass geplante Konnektoren bereits live sind.

Die wichtigste Erfahrung

28 grüne Tests übersahen, dass altes und neues System jeweils einen Timer starten konnten. Reviews fanden diesen Fehler und später einen engen Schema-6-Migrationsfall. Die finale Version erreichte Schema 7 und 35 Tests.

Danach zeigte die menschliche Abnahme ein einfacheres Problem: Hilfstexte waren zu klein. Obwohl Wiederherstellung und mobile Breite geprüft waren, war die Seite anstrengend zu lesen. Text und Aufgaben wurden auf 18px, Labels auf 15px und Bedienelemente auf 17px mit 48px Höhe vergrößert.

Automatisierung beweist, dass Software ihre Regeln einhält. Der menschliche Körper beweist, dass das Produkt tatsächlich benutzbar ist.

Fazit

Der eigentliche Wert ist eine vertrauenswürdige Verantwortungskette durch den Tag. Sie kostet mehr Code, einen langsameren ersten Release und strengere Gewohnheiten. Für langfristiges Lernen über die eigene Zeit ist dieser Preis ehrlicher als ein einfacheres Werkzeug, das seine Geschichte unbemerkt verfälscht.
