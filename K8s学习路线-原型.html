容器与 Docker 基础

Container & Docker Grundlagen

Container (容器) - 轻量级虚拟化

概念：容器是独立的运行环境，包含应用代码、运行时、系统工具、系统库。

为什么需要？

  • ✅ 环境一致性：开发、测试、生产环境完全一致（"在我机器上可以跑"问题彻底解决）
  • ✅ 资源隔离：进程、网络、文件系统隔离（但共享宿主机内核，比虚拟机轻量）
  • ✅ 快速启动：秒级启动（虚拟机需要分钟级）
  • ✅ 高密度部署：一台物理机可以运行几十个容器（虚拟机只能几个）

实际例子：你开发一个 Node.js 应用，本地用 Node 18，生产环境用 Node 16，可能出现兼容性问题。用容器，你可以把 Node 18 和运行环境一起打包，任何地方运行都是 Node 18 环境。

Konzept: Ein Container ist eine isolierte Laufzeitumgebung, die Anwendungscode, Laufzeit, Systemwerkzeuge und Systembibliotheken enthält.

Warum wird es benötigt?

  • ✅ Umgebungskonsistenz: Entwicklungs-, Test- und Produktionsumgebungen sind identisch (das Problem "auf meinem Rechner läuft es" ist vollständig gelöst)
  • ✅ Ressourcenisolation: Prozess-, Netzwerk- und Dateisystemisolation (teilt sich aber den Host-Kernel, leichtgewichtiger als VM)
  • ✅ Schneller Start: Sekundenbereich (VMs brauchen Minuten)
  • ✅ Hohe Dichte-Bereitstellung: Ein physischer Rechner kann Dutzende Container ausführen (VMs nur wenige)

Praxis-Beispiel: Sie entwickeln eine Node.js-Anwendung, lokal verwenden Sie Node 18, in der Produktionsumgebung wird Node 16 verwendet, es können Kompatibilitätsprobleme auftreten. Mit Containern können Sie Node 18 zusammen mit der Laufzeitumgebung verpacken, und überall wird die Node 18-Umgebung ausgeführt.

Docker Image (镜像) - 不可变的模板

概念：镜像是只读模板，用于创建容器。类似"软件安装包"，但是分层存储（每层只读，可复用）。

分层存储的优势：

  • ✅ 复用：多个镜像可以共享基础层（比如多个 Node.js 应用都基于 node:18 镜像）
  • ✅ 增量传输：更新镜像只需传输变化的层（节省带宽和时间）
  • ✅ 版本管理：每层都有哈希值，可以精确回滚
💡 实战案例：你的应用基于 node:18，大小 200MB。如果你有 10 个应用都基于 node:18，Docker 只需要存储一份 node:18 层，所有容器共享它。总存储 = 200MB + 10 * (你的应用大小)。

Konzept: Ein Image ist eine schreibgeschützte Vorlage zur Erstellung von Containern. Ähnlich wie ein "Software-Installationspaket", aber mit Schichtenspeicher (jede Schicht ist schreibgeschützt und wiederverwendbar).

Vorteile des Schichtenspeichers:

  • ✅ Wiederverwendbarkeit: Mehrere Images können Basisschichten teilen (z.B. mehrere Node.js-Anwendungen basieren alle auf dem node:18-Image)
  • ✅ Inkrementelle Übertragung: Bei Image-Updates müssen nur die sich ändernden Schichten übertragen werden (spart Bandbreite und Zeit)
  • ✅ Versionsverwaltung: Jede Schicht hat einen Hash-Wert, ermöglicht präzises Rollback
💡 Praxis-Beispiel: Ihre Anwendung basiert auf node:18, Größe 200MB. Wenn Sie 10 Anwendungen haben, die alle auf node:18 basieren, muss Docker nur eine Kopie der node:18-Schicht speichern, alle Container teilen sich diese. Gesamtspeicher = 200MB + 10 * (Größe Ihrer Anwendung).

Dockerfile - 构建镜像的配方

概念：Dockerfile 是文本文件，包含构建镜像的指令。类似"菜谱"，一步步告诉 Docker 如何构建镜像。

# 多阶段构建示例（优化镜像体积）
# 阶段 1：构建
FROM node:18 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# 阶段 2：运行（最终镜像仅 80MB）
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
EXPOSE 80

✅ 多阶段构建优势

  • 镜像体积小（去掉构建工具）
  • 安全性高（减少攻击面）
  • 适合生产环境

❌ 单阶段构建

  • 构建速度快
  • 调试方便
  • 适合开发环境

Konzept: Ein Dockerfile ist eine Textdatei, die Anweisungen zum Erstellen eines Images enthält. Ähnlich wie ein "Rezept", das Docker Schritt für Schritt erklärt, wie das Image erstellt wird.

# Multi-Stage-Build-Beispiel (optimiert Image-Größe)
# Phase 1: Build
FROM node:18 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# Phase 2: Ausführung (finales Image nur 80MB)
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
EXPOSE 80

✅ Vorteile Multi-Stage-Build

  • Kleine Image-Größe (Build-Tools entfernt)
  • Hohe Sicherheit (reduzierte Angriffsfläche)
  • Geeignet für Produktionsumgebung

❌ Single-Stage-Build

  • Schnelleres Build
  • Einfacheres Debugging
  • Geeignet für Entwicklungsumgebung

Kubernetes 架构核心

Kubernetes Architektur-Kern

Control Plane (控制平面) - 集群的大脑

概念：Control Plane 是 K8s 集群的管理中心，负责决策和调度。

核心组件：

  • API Server：统一入口（所有操作都通过它，类似"前台"）
  • etcd：分布式键值存储（保存集群状态，类似"数据库"）
  • Scheduler：调度器（决定 Pod 运行在哪个 Node，考虑资源、亲和性等）
  • Controller Manager：控制器管理器（确保实际状态匹配期望状态，比如副本数不够就创建新 Pod）
💡 实际例子：你创建一个 Deployment（期望 3 个 Pod 副本）。API Server 接收请求并写入 etcd。Controller Manager 发现实际副本数是 0，小于期望的 3。于是它创建 3 个 Pod 对象。Scheduler 看到未调度的 Pod，根据资源情况分配到合适的 Node。

Konzept: Die Control Plane ist das Verwaltungszentrum des K8s-Clusters und verantwortlich für Entscheidungen und Scheduling.

Kernkomponenten:

  • API Server: Zentraler Eingang (alle Operationen laufen über ihn, ähnlich wie eine "Rezeption")
  • etcd: Verteilter Key-Value-Speicher (speichert Cluster-Status, ähnlich einer "Datenbank")
  • Scheduler: Scheduler (entscheidet, auf welchem Node Pods ausgeführt werden, berücksichtigt Ressourcen, Affinität usw.)
  • Controller Manager: Controller-Manager (stellt sicher, dass der tatsächliche Status mit dem gewünschten Status übereinstimmt, z.B. erstellt neue Pods, wenn die Replikatanzahl unzureichend ist)
💡 Praxis-Beispiel: Sie erstellen ein Deployment (erwartet 3 Pod-Replikate). Der API Server empfängt die Anfrage und schreibt in etcd. Der Controller Manager stellt fest, dass die tatsächliche Replikatanzahl 0 ist, was geringer als die erwartete 3 ist. Also erstellt er 3 Pod-Objekte. Der Scheduler sieht nicht geplante Pods und weist sie basierend auf der Ressourcensituation geeigneten Nodes zu.

Worker Nodes (工作节点) - 干活的"手"

概念：Worker Nodes 是实际运行容器的机器（物理机或虚拟机）。

核心组件：

  • Kubelet：Node 的"代理人"（与 API Server 通信，管理 Pod 生命周期）
  • Kube-proxy：网络代理（实现 Service 的抽象，负责流量转发）
  • Container Runtime：容器运行时（真正运行容器的软件，如 containerd、Docker）

为什么需要 Kubelet？ API Server 不直接管理容器，它通过 Kubelet 间接管理。Kubelet 监听 API Server 的事件，然后根据指令操作本地的容器运行时。

Konzept: Worker Nodes sind die Maschinen (physisch oder virtuell), auf denen tatsächlich Container ausgeführt werden.

Kernkomponenten:

  • Kubelet: "Agent" des Nodes (kommuniziert mit dem API Server, verwaltet Pod-Lebenszyklus)
  • Kube-proxy: Netzwerk-Proxy (implementiert die Service-Abstraktion, verantwortlich für Datenverkehrsweiterleitung)
  • Container Runtime: Container-Laufzeitumgebung (Software, die tatsächlich Container ausführt, wie containerd, Docker)

Warum wird Kubelet benötigt? Der API Server verwaltet Container nicht direkt, er verwaltet sie indirekt über Kubelet. Kubelet überwacht Ereignisse vom API Server und führt dann Operationen auf der lokalen Container-Laufzeitumgebung basierend auf den Anweisungen aus.

Declarative API (声明式 API) - K8s 的核心理念

概念：你告诉 K8s "我想要什么状态"（期望状态），K8s 自动实现它（通过控制循环）。

对比：

命令式 (Imperative) 声明式 (Declarative)
kubectl run nginx --image=nginx kubectl apply -f deployment.yaml
"创建一个 Nginx 容器" "确保有 3 个 Nginx Pod 在运行"
关注"如何做" 关注"做什么"
💡 实际例子：你声明"我想要 3 个 Pod 副本"。如果某个 Node 故障，K8s 自动在其他 Node 上创建新的 Pod，始终保持 3 个副本。你不需要手动干预，K8s 自己"干活"。

Konzept: Sie sagen K8s "welchen Status ich will" (gewünschter Status), und K8s erreicht dies automatisch (durch Control Loop).

Vergleich:

Imperativ Deklarativ
kubectl run nginx --image=nginx kubectl apply -f deployment.yaml
"Erstelle einen Nginx-Container" "Stellen Sie sicher, dass 3 Nginx Pods ausgeführt werden"
Konzentriert sich auf "wie es getan wird" Konzentriert sich auf "was getan wird"
💡 Praxis-Beispiel: Sie deklarieren "Ich möchte 3 Pod-Replikate". Wenn ein Node ausfällt, erstellt K8s automatisch neue Pods auf anderen Nodes und hält immer 3 Replikate aufrecht. Sie müssen nicht manuell eingreifen, K8s "arbeitet" selbstständig.
