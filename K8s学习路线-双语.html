Grundlagen-Vorbereitungsphase

Linux 基础 (Linux Fundamentals)

为什么重要：K8s 底层运行在 Linux 上，需要熟悉命令行、进程管理、网络基础。

核心知识点：

  • 文件系统结构 (Filesystem Hierarchy)
  • 进程管理 (Process Management) - ps, top, systemd
  • 网络基础 (Networking Basics) - IP, DNS, iptables
  • 权限管理 (Permission Management) - user, group, chmod

📝 练习任务

在本地虚拟机或云服务器上：

  • 熟练使用 vim / nano 编辑器
  • 配置 SSH 免密登录
  • 设置防火墙规则 (iptables / firewalld)
  • 查看系统资源使用情况 (df, free, top)

Linux Grundlagen (Linux Fundamentals)

Warum wichtig: K8s läuft unter Linux, vertraute Kenntnisse in Kommandozeile, Prozessverwaltung und Netzwerkgrundlagen sind erforderlich.

Kernwissenspunkte:

  • Dateisystem-Hierarchie (Filesystem Hierarchy)
  • Prozessverwaltung (Process Management) - ps, top, systemd
  • Netzwerkgrundlagen (Networking Basics) - IP, DNS, iptables
  • Berechtigungsverwaltung (Permission Management) - user, group, chmod

📝 Übungsaufgaben

Auf lokaler VM oder Cloud-Server:

  • Beherrschen von vim / nano Editoren
  • SSH-Passwortfreie Anmeldung konfigurieren
  • Firewall-Regeln einrichten (iptables / firewalld)
  • Systemressourcennutzung prüfen (df, free, top)

Docker 基础 (Docker Fundamentals)

核心概念：容器化技术的基石，K8s 的底层运行时。

🏗️ Docker 架构 (Docker Architecture)

Client (Docker CLI) ↓ Docker Daemon (dockerd) ↓ Container Runtime (containerd / runc) ↓ Containers (隔离的进程空间)

必须掌握：

  • 镜像 (Image)：只读模板，分层存储 (Layer Storage)
  • 容器 (Container)：镜像的运行实例，轻量级虚拟化
  • Dockerfile：构建镜像的指令文件
  • Registry：镜像存储仓库 (如 Docker Hub)
💡 实际例子： 构建一个 Node.js 应用镜像 
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install --production
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]

⚖️ 架构取舍：镜像大小 vs 构建速度

场景：多阶段构建 (Multi-stage Build) 的选择

✅ 多阶段构建
  • 最终镜像体积小 (去掉构建工具)
  • 安全性更高 (减少攻击面)
  • 适合生产环境
❌ 单阶段构建
  • 构建速度快 (无需多步)
  • 调试方便 (包含完整工具)
  • 适合开发环境

实践建议：开发用单阶段（快速迭代），生产用多阶段（优化体积）。

📝 练习任务

  • 安装 Docker Desktop (Mac/Windows) 或 Docker Engine (Linux)
  • 拉取并运行 Nginx 镜像：docker run -d -p 80:80 nginx
  • 编写 Dockerfile 构建自定义镜像
  • 使用 Docker Compose 编排多容器应用
  • 推送镜像到 Docker Hub 或私有 Registry

Docker Grundlagen (Docker Fundamentals)

Kernkonzepte: Grundlage der Containerisierungstechnologie, unterliegende Laufzeitumgebung von K8s.

🏗️ Docker-Architektur (Docker Architecture)

Client (Docker CLI) ↓ Docker Daemon (dockerd) ↓ Container Runtime (containerd / runc) ↓ Containers (Isolierter Prozessraum)

Unbedingt beherrschen:

  • Image (Abbild): Schreibgeschützte Vorlage, Schichtenspeicher (Layer Storage)
  • Container: Laufende Instanz eines Images, leichtgewichtige Virtualisierung
  • Dockerfile: Anweisungsdatei zum Erstellen eines Images
  • Registry: Image-Speicher-Repository (z.B. Docker Hub)
💡 Praxis-Beispiel: Ein Node.js-Anwendungs-Image erstellen 
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install --production
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]

⚖️ Architektur-Trade-off: Image-Größe vs Build-Geschwindigkeit

Szenario: Multi-Stage-Build-Auswahl

✅ Multi-Stage-Build
  • Finales Image klein (Build-Tools entfernt)
  • Höhere Sicherheit (reduzierte Angriffsfläche)
  • Geeignet für Produktionsumgebung
❌ Single-Stage-Build
  • Schnellerer Build (keine mehrstufigen Schritte)
  • Einfacheres Debugging (enthält alle Tools)
  • Geeignet für Entwicklungsumgebung

Praxis-Empfehlung: Entwicklung verwendet Single-Stage (schnelle Iteration), Produktion verwendet Multi-Stage (optimierte Größe).

📝 Übungsaufgaben

  • Docker Desktop (Mac/Windows) oder Docker Engine (Linux) installieren
  • Nginx-Image pullen und ausführen: docker run -d -p 80:80 nginx
  • Dockerfile schreiben, um benutzerdefiniertes Image zu erstellen
  • Docker Compose verwenden, um Multi-Container-Anwendungen zu orchestrieren
  • Image zu Docker Hub oder privater Registry pushen

AWS 云平台基础

AWS Cloud-Plattform Grundlagen

AWS 核心服务 (AWS Core Services)

为什么重要：AWS 是 K8s 最常见的运行环境 (EKS)，需要理解云基础设施。

🏗️ AWS VPC 网络架构 (VPC Networking Architecture)

VPC (Virtual Private Cloud) - 10.0.0.0/16 ├── Public Subnet (10.0.1.0/24) │ ├── Internet Gateway (IGW) │ └── NAT Gateway ├── Private Subnet (10.0.2.0/24) │ └── EC2 Instances / EKS Worker Nodes └── Private Subnet (10.0.3.0/24) └── Database (RDS / Aurora)

核心服务：

  • EC2 (Elastic Compute Cloud)：虚拟服务器，K8s 节点的运行载体
  • VPC (Virtual Private Cloud)：隔离的虚拟网络环境
  • EBS (Elastic Block Store)：持久化块存储，类似硬盘
  • ELB (Elastic Load Balancing)：负载均衡器，流量入口
  • IAM (Identity and Access Management)：权限管理，安全控制
  • S3 (Simple Storage Service)：对象存储，存镜像/日志/备份
💡 实际例子： 在 AWS 上手动部署 K8s 前的网络规划
  • 创建 VPC：10.0.0.0/16 (65536 个 IP)
  • 公有子网：10.0.1.0/24 (256 个 IP，放 ELB)
  • 私有子网：10.0.2.0/24 (256 个 IP，放 Worker Nodes)
  • 配置 NAT Gateway：让私有子网访问互联网 (下载镜像)

⚖️ 架构取舍：自动驾驶数据处理流水线架构

实际场景：激光雷达点云数据实时处理 + 仿真环境多节点部署

✅ 集中式流处理架构 (Fluentd → Kafka → Flink)
  • 低延迟实时处理 (< 100ms)
  • 适合自动驾驶感知流水线
  • Kafka 确保数据不丢失
  • Flink 状态管理适合复杂聚合
❌ 分布式批处理架构 (S3 → Spark)
  • 高吞吐量离线分析
  • 适合仿真回放数据
  • 成本低 (按需启动)
  • 延迟高 (分钟级)

实践建议：生产用流处理 (实时感知)，开发用批处理 (仿真回放)。LiangDao 采用混合架构：流处理负责实时数据，批处理负责离线仿真分析。

📝 练习任务

  • 注册 AWS 账号 (需要信用卡，有免费套餐)
  • 启动一个 EC2 实例 (t2.micro，免费套餐)
  • 创建 VPC 和子网，配置安全组 (Security Group)
  • 挂载 EBS 卷到 EC2，测试持久化存储
  • 配置 IAM 用户和角色，理解最小权限原则

AWS Kerndienste (AWS Core Services)

Warum wichtig: AWS ist die häufigste K8s-Laufzeitumgebung (EKS), Cloud-Infrastruktur muss verstanden werden.

🏗️ AWS VPC-Netzwerkarchitektur (VPC Networking Architecture)

VPC (Virtual Private Cloud) - 10.0.0.0/16 ├── Public Subnet (10.0.1.0/24) │ ├── Internet Gateway (IGW) │ └── NAT Gateway ├── Private Subnet (10.0.2.0/24) │ └── EC2 Instances / EKS Worker Nodes └── Private Subnet (10.0.3.0/24) └── Datenbank (RDS / Aurora)

Kerndienste:

  • EC2 (Elastic Compute Cloud): Virtueller Server, Träger für K8s-Nodes
  • VPC (Virtual Private Cloud): Isolierte virtuelle Netzwerkumgebung
  • EBS (Elastic Block Store): Persistenter Blockspeicher, ähnlich einer Festplatte
  • ELB (Elastic Load Balancing): Load Balancer, Datenverkehrseingang
  • IAM (Identity and Access Management): Berechtigungsverwaltung, Sicherheitskontrolle
  • S3 (Simple Storage Service): Objektspeicher, Images/Logs/Backups speichern
💡 Praxis-Beispiel: Netzwerkplanung vor manueller K8s-Bereitstellung auf AWS
  • VPC erstellen: 10.0.0.0/16 (65536 IPs)
  • Öffentliches Subnetz: 10.0.1.0/24 (256 IPs, für ELB)
  • Privates Subnetz: 10.0.2.0/24 (256 IPs, für Worker Nodes)
  • NAT Gateway konfigurieren: Privatem Subnetz Internetzugang ermöglichen (Images herunterladen)

⚖️ Architektur-Trade-off: Autonomes Fahren Datenverarbeitungs-Pipeline

Praxis-Szenario: LiDAR Point-Cloud-Daten Echtzeitverarbeitung + Simulationsumgebung Multi-Node-Bereitstellung

✅ Zentralisierte Stream-Verarbeitung (Fluentd → Kafka → Flink)
  • Geringe Latenz Echtzeitverarbeitung (< 100ms)
  • Geeignet für autonomes Fahren Perzeption-Pipeline
  • Kafka garantiert keine Datenverluste
  • Flink Statusmanagement für komplexe Aggregation
❌ Verteilte Batch-Verarbeitung (S3 → Spark)
  • Hoher Durchsatz für Offline-Analyse
  • Geeignet für Simulations-Wiedergabedaten
  • Niedrige Kosten (bedarfsgesteuert)
  • Höhere Latenz (Minutenebene)

Praxis-Empfehlung: Produktion: Stream-Verarbeitung (Echtzeit-Perzeption), Entwicklung: Batch-Verarbeitung (Simulations-Wiedergabe). LiangDao verwendet Hybrid-Architektur: Stream für Echtzeitdaten, Batch für Offline-Simulationsanalyse.

📝 Übungsaufgaben

  • AWS-Konto registrieren (Kreditkarte erforderlich, kostenloses Kontingent verfügbar)
  • Eine EC2-Instanz starten (t2.micro, kostenloses Kontingent)
  • VPC und Subnetze erstellen, Security Groups konfigurieren
  • EBS-Volume an EC2 anhängen, persistenten Speicher testen
  • IAM-Benutzer und -Rollen konfigurieren, Prinzip der geringsten Rechte verstehen

Kubernetes 核心概念

Kubernetes Kernkonzepte

K8s 架构 (Kubernetes Architecture)

🏗️ K8s 集群架构 (Cluster Architecture)

Control Plane (Master Node) - 大脑 ├── API Server (kube-apiserver) - 统一入口 ├── etcd - 分布式键值存储 (集群状态) ├── Scheduler (kube-scheduler) - 调度 Pod 到 Node ├── Controller Manager (kube-controller-manager) - 维护期望状态 └── Cloud Controller Manager - 对接云平台 (可选) Worker Nodes - 干活的手 ├── Kubelet - 与 API Server 通信，管理 Pod ├── Kube-proxy - 网络代理，Service 实现 ├── Container Runtime (containerd / Docker) - 运行容器 └── Pods - 最小调度单元 (包含 1+ 容器)

核心原理：

  • 声明式 API (Declarative API)：告诉 K8s "想要什么状态"，而不是"怎么做"
  • 控制循环 (Control Loop)：不断对比 "当前状态" 和 "期望状态"，自动修复
  • 松耦合 (Loose Coupling)：各组件通过 API Server 通信，互不依赖
💡 实际例子： 副本自愈 (Self-healing)

你声明：replicas: 3 (想要 3 个 Pod)

某个 Pod 崩溃 → K8s 检测到 "当前 2 个" ≠ "期望 3 个" → 自动创建新 Pod

对比传统：需要手动 restart，或写 Supervisor 脚本。

K8s Architektur (Kubernetes Architecture)

🏗️ K8s-Cluster-Architektur (Cluster Architecture)

Control Plane (Master Node) - Gehirn ├── API Server (kube-apiserver) - Zentraler Eingang ├── etcd - Verteilter Key-Value-Speicher (Cluster-Status) ├── Scheduler (kube-scheduler) - Plant Pods auf Nodes ├── Controller Manager (kube-controller-manager) - Hält gewünschten Status aufrecht └── Cloud Controller Manager - Cloud-Plattform-Anbindung (optional) Worker Nodes - Arbeitende Hände ├── Kubelet - Kommuniziert mit API Server, verwaltet Pods ├── Kube-proxy - Netzwerk-Proxy, Service-Implementierung ├── Container Runtime (containerd / Docker) - Führt Container aus └── Pods - Kleinste Planungseinheit (enthält 1+ Container)

Kernprinzipien:

  • Deklarative API (Declarative API): K8s mitteilen "welchen Status ich will", nicht "wie es gemacht wird"
  • Control Loop (Kontrollschleife): Ständiger Vergleich von "Ist-Zustand" und "Soll-Zustand", automatische Korrektur
  • Lose Kopplung (Loose Coupling): Komponenten kommunizieren über API Server, keine gegenseitigen Abhängigkeiten
💡 Praxis-Beispiel: Replikat-Selbstheilung (Self-healing)

Sie deklarieren: replicas: 3 (3 Pods erwartet)

Ein Pod stürzt ab → K8s erkennt "aktuell 2" ≠ "erwartet 3" → Erstellt automatisch neuen Pod

Im Vergleich zu traditionell: Manuelles Restart erforderlich oder Supervisor-Skript schreiben.

Pod - 最小调度单元 (Pod - Smallest Deployable Unit)

为什么需要 Pod：容器共享网络/存储，类似 "豌豆荚" 的概念。

🏗️ Pod 内部结构 (Pod Internal Structure)

Pod (IP: 10.244.1.5) ├── Pause Container (基础设施容器，持有网络命名空间) ├── App Container (你的应用，如 Nginx) └── Sidecar Container (辅助容器，如日志收集)

关键特性：

  • 共享网络 (Shared Network)：Pod 内所有容器共享 IP 和端口空间
  • 共享存储 (Shared Storage)：Volume 可以被多个容器挂载
  • 生命周期 (Lifecycle)：Pod 是短暂的，随时可能被重建

⚖️ 架构取舍：一个 Pod 运行多个容器 vs 每个 Pod 一个容器

✅ Sidecar 模式 (多容器 Pod)
  • 紧密耦合的服务 (如日志收集)
  • 共享存储和网络 (高效通信)
  • 统一生命周期 (一起扩缩容)
❌ 单容器 Pod
  • 独立扩缩容 (精细化资源控制)
  • 故障隔离 (一个挂了不影响其他)
  • 符合微服务原则 (单一职责)

实践建议：除非是 Sidecar (如 Istio Proxy)，否则一个 Pod 只放一个容器。

📝 练习任务

  • 安装 Minikube (本地 K8s 单机版)
  • 运行第一个 Pod：kubectl run nginx --image=nginx
  • 查看 Pod 详情：kubectl describe pod nginx
  • 进入 Pod 内部：kubectl exec -it nginx -- /bin/bash
  • 删除 Pod 观察重建 (配合 Deployment)

Pod - Kleinste Planungseinheit (Pod - Smallest Deployable Unit)

Warum Pod benötigt: Container teilen Netzwerk/Speicher, ähnlich dem Konzept einer "Erbse in einer Hülse".

🏗️ Pod-Interne Struktur (Pod Internal Structure)

Pod (IP: 10.244.1.5) ├── Pause Container (Infrastruktur-Container, hält Netzwerk-Namensraum) ├── App Container (Ihre Anwendung, z.B. Nginx) └── Sidecar Container (Hilfs-Container, z.B. Log-Sammlung)

Wichtige Eigenschaften:

  • Geteiltes Netzwerk (Shared Network): Alle Container im Pod teilen IP und Port-Raum
  • Geteilter Speicher (Shared Storage): Volume kann von mehreren Containern eingehängt werden
  • Lebenszyklus (Lifecycle): Pods sind kurzlebig, können jederzeit neu erstellt werden

⚖️ Architektur-Trade-off: Mehrere Container in einem Pod vs Ein Container pro Pod

✅ Sidecar-Muster (Multi-Container Pod)
  • Eng gekoppelte Dienste (z.B. Log-Sammlung)
  • Geteilter Speicher und Netzwerk (effiziente Kommunikation)
  • Einheitlicher Lebenszyklus (zusammen skalieren)
❌ Single-Container Pod
  • Unabhängige Skalierung (feinkörnige Ressourcenkontrolle)
  • Fehlerisolation (einer fällt aus, andere nicht betroffen)
  • Entspricht Microservice-Prinzipien (Single Responsibility)

Praxis-Empfehlung: Außer bei Sidecars (z.B. Istio Proxy) sollte ein Pod nur einen Container enthalten.

📝 Übungsaufgaben

  • Minikube installieren (lokale K8s-Einplatzversion)
  • Ersten Pod ausführen: kubectl run nginx --image=nginx
  • Pod-Details anzeigen: kubectl describe pod nginx
  • In Pod eintreten: kubectl exec -it nginx -- /bin/bash
  • Pod löschen und Neuerstellung beobachten (mit Deployment)

工作负载控制器 (Workload Controllers)

为什么需要控制器：直接管理 Pod 太累，控制器提供声明式管理和自愈能力。

核心控制器对比：

  • Deployment：无状态应用 (Stateless)，支持滚动更新 (Rolling Update)
  • StatefulSet：有状态应用 (Stateful)，稳定的网络标识和存储
  • DaemonSet：每个 Node 运行一个副本 (如日志收集器)
  • Job / CronJob：批处理任务 (Batch Jobs)
💡 实际例子： Deployment 的滚动更新策略 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 最多多 1 个 Pod
      maxUnavailable: 1   # 最多少 1 个 Pod

⚖️ 架构取舍：Deployment vs StatefulSet

✅ Deployment (无状态)
  • Pod 名称随机 (nginx-xxxx-xxx)
  • 扩缩容快 (无需顺序)
  • 适合 Web 服务、API
✅ StatefulSet (有状态)
  • Pod 名称固定 (mysql-0, mysql-1)
  • 稳定的网络标识 (Persistent Identity)
  • 适合数据库、消息队列

实践建议：99% 的场景用 Deployment，只有需要稳定标识时才用 StatefulSet。

Workload-Controller (Workload Controllers)

Warum Controller benötigt: Direkte Pod-Verwaltung ist zu mühsam, Controller bieten deklarative Verwaltung und Selbstheilungsfähigkeiten.

Kern-Controller-Vergleich:

  • Deployment: Stateless Apps, Rolling Update unterstützt
  • StatefulSet: Stateful Apps, stabile Netzwerkidentität und Speicher
  • DaemonSet: Ein Replikat pro Node (z.B. Log-Sammler)
  • Job / CronJob: Batch-Aufgaben
💡 Praxis-Beispiel: Rolling-Update-Strategie von Deployment 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # Maximal 1 zusätzlicher Pod
      maxUnavailable: 1   # Maximal 1 Pod weniger

⚖️ Architektur-Trade-off: Deployment vs StatefulSet

✅ Deployment (zustandslos)
  • Zufällige Pod-Namen (nginx-xxxx-xxx)
  • Schnelle Skalierung (keine Reihenfolge erforderlich)
  • Geeignet für Web-Services, APIs
✅ StatefulSet (zustandsbehaftet)
  • Feste Pod-Namen (mysql-0, mysql-1)
  • Stabile Netzwerkidentität (Persistent Identity)
  • Geeignet für Datenbanken, Message Queues

Praxis-Empfehlung: 99% der Szenarien verwenden Deployment, nur wenn stabile Identität erforderlich ist.

网络与存储

Netzwerk & Speicher

K8s 网络模型 (Kubernetes Networking Model)

核心原则：所有 Pod 可以互相通信，所有 Pod 可以与所有 Node 通信。

🏗️ K8s 网络架构 (Network Architecture)

Service (虚拟 IP，负载均衡) ↓ Pod (10.244.1.5) ←→ Pod (10.244.2.3) [Pod 间通信] ↓ Node (物理机) ↓ Internet / External Service

三层网络模型：

  • Container-to-Container (同一 Pod)：localhost 通信
  • Pod-to-Pod (跨 Node)：Overlay 网络 (如 Flannel, Calico)
  • Pod-to-Service：通过 kube-proxy 实现负载均衡
  • External-to-Service：NodePort / LoadBalancer / Ingress
💡 实际例子： Service 的负载均衡 
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: ClusterIP
  selector:
    app: nginx
  ports:
    - protocol: TCP
      port: 80
      targetPort: 80

⚖️ 架构取舍：Service 类型选择

✅ ClusterIP (默认)
  • 仅在集群内可访问
  • 适合后端服务 (如数据库)
  • 安全 (不暴露到外网)
✅ LoadBalancer (生产推荐)
  • 自动创建云平台负载均衡器
  • 高可用 (跨可用区)
  • 成本高 (每个 Service 一个 LB)

实践建议：外部流量用 Ingress (七层) + LoadBalancer (四层)，内部通信用 ClusterIP。

K8s-Netzwerkmodell (Kubernetes Networking Model)

Kernprinzip: Alle Pods können miteinander kommunizieren, alle Pods können mit allen Nodes kommunizieren.

🏗️ K8s-Netzwerkarchitektur (Network Architecture)

Service (Virtuelle IP, Load Balancing) ↓ Pod (10.244.1.5) ←→ Pod (10.244.2.3) [Pod-zu-Pod-Kommunikation] ↓ Node (Physische Maschine) ↓ Internet / Externer Service

Drei-Ebenen-Netzwerkmodell:

  • Container-zu-Container (gleicher Pod): localhost-Kommunikation
  • Pod-zu-Pod (über Node-Grenzen): Overlay-Netzwerk (z.B. Flannel, Calico)
  • Pod-zu-Service: Load Balancing durch kube-proxy
  • Extern-zu-Service: NodePort / LoadBalancer / Ingress
💡 Praxis-Beispiel: Service Load Balancing 
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: ClusterIP
  selector:
    app: nginx
  ports:
    - protocol: TCP
      port: 80
      targetPort: 80

⚖️ Architektur-Trade-off: Service-Typ-Auswahl

✅ ClusterIP (Standard)
  • Nur innerhalb des Clusters erreichbar
  • Geeignet für Backend-Dienste (z.B. Datenbanken)
  • Sicher (nicht nach außen exponiert)
✅ LoadBalancer (Produktion empfohlen)
  • Erstellt automatisch Cloud-Plattform Load Balancer
  • Hochverfügbar (über Verfügbarkeitszonen)
  • Höhere Kosten (ein LB pro Service)

Praxis-Empfehlung: Externer Datenverkehr: Ingress (Layer 7) + LoadBalancer (Layer 4), Interne Kommunikation: ClusterIP.

持久化存储 (Persistent Storage)

问题：Pod 是短暂的，数据不能存在容器里 (重启就没了)。

核心概念：

  • PV (PersistentVolume)：集群级别的存储资源 (由管理员创建)
  • PVC (PersistentVolumeClaim)：用户对存储的请求 (类似 "下单")
  • StorageClass：动态供应 (Dynamic Provisioning)，自动创建 PV
💡 实际例子： AWS EBS 作为持久化存储 
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: gp3-storage
provisioner: kubernetes.io/aws-ebs
parameters:
  type: gp3
  fsType: ext4

⚖️ 架构取舍：存储访问模式

✅ ReadWriteOnce (RWO)
  • 只能被一个 Node 挂载
  • 性能最好 (块存储，如 EBS)
  • 适合数据库
✅ ReadWriteMany (RWX)
  • 可以被多个 Node 同时挂载
  • 适合共享文件系统 (如 NFS, EFS)
  • 性能较差 (网络文件系统)

实践建议：数据库用 RWO (EBS)，共享配置用 RWX (EFS/NFS)。

Persistenter Speicher (Persistent Storage)

Problem: Pods sind kurzlebig, Daten können nicht im Container gespeichert werden (nach Neustart weg).

Kernkonzepte:

  • PV (PersistentVolume): Clusterspeicherressource (von Administrator erstellt)
  • PVC (PersistentVolumeClaim): Speicheranfrage des Benutzers (ähnlich "Bestellung")
  • StorageClass: Dynamische Bereitstellung, automatische PV-Erstellung
💡 Praxis-Beispiel: AWS EBS als persistenter Speicher 
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: gp3-storage
provisioner: kubernetes.io/aws-ebs
parameters:
  type: gp3
  fsType: ext4

⚖️ Architektur-Trade-off: Speicherzugriffsmodi

✅ ReadWriteOnce (RWO)
  • Kann nur von einem Node eingehängt werden
  • Beste Leistung (Block-Speicher, z.B. EBS)
  • Geeignet für Datenbanken
✅ ReadWriteMany (RWX)
  • Kann gleichzeitig von mehreren Nodes eingehängt werden
  • Geeignet für gemeinsame Dateisysteme (z.B. NFS, EFS)
  • Schlechtere Leistung (Netzwerk-Dateisystem)

Praxis-Empfehlung: Datenbanken: RWO (EBS), gemeinsame Konfiguration: RWX (EFS/NFS).

配置管理与安全

Konfigurationsverwaltung & Sicherheit

ConfigMap 与 Secret (Configuration & Secrets Management)

为什么需要：将配置与代码分离 (12-Factor App 原则)。

核心区别：

  • ConfigMap：存储非敏感配置 (如配置文件、环境变量)
  • Secret：存储敏感信息 (如密码、API Key)，base64 编码 (不是加密！)
💡 实际例子： 使用 ConfigMap 注入配置 
kubectl create configmap app-config --from-file=app.conf

⚖️ 架构取舍：配置注入方式

✅ 环境变量 (env)
  • 简单直接
  • 适合简单配置 (如 API_URL)
  • 应用无需修改代码
✅ 文件挂载 (volumeMount)
  • 支持复杂配置 (如 Nginx.conf)
  • 支持动态更新
  • 适合配置文件

实践建议：简单 K/V 用环境变量，配置文件用挂载。

📝 练习任务

  • 在 Pod 中使用配置
  • 使用 kubectl edit 修改 ConfigMap，观察应用是否自动更新
  • 集成 AWS Secrets Manager

ConfigMap & Secret (Configuration & Secrets Management)

Warum benötigt: Konfiguration von Code trennen (12-Factor App Prinzip).

Kernunterschiede:

  • ConfigMap: Speichert nicht-sensible Konfiguration (z.B. Konfigurationsdateien, Umgebungsvariablen)
  • Secret: Speichert sensible Informationen (z.B. Passwörter, API-Keys), base64-kodiert (nicht verschlüsselt!)
💡 Praxis-Beispiel: ConfigMap zur Konfigurationsinjektion verwenden 
kubectl create configmap app-config --from-file=app.conf

⚖️ Architektur-Trade-off: Konfigurationsinjektionsmethode

✅ Umgebungsvariablen (env)
  • Einfach und direkt
  • Geeignet für einfache Konfiguration (z.B. API_URL)
  • Anwendung muss Code nicht ändern
✅ Dateieinbindung (volumeMount)
  • Unterstützt komplexe Konfiguration (z.B. Nginx.conf)
  • Unterstützt dynamische Updates
  • Geeignet für Konfigurationsdateien

Praxis-Empfehlung: Einfache K/V: Umgebungsvariablen, Konfigurationsdateien: Einbindung.

📝 Übungsaufgaben

  • Konfiguration in Pods verwenden
  • Mit kubectl edit ConfigMap ändern, beobachten ob Anwendung automatisch aktualisiert

Ingress - 七层路由 (Ingress - L7 Routing)

为什么需要：Service (LoadBalancer) 每个服务需要一个 LB，太贵！Ingress 提供 HTTP/HTTPS 路由。

🏗️ Ingress 架构 (Ingress Architecture)

Internet ↓ LoadBalancer (AWS ELB) - 1 个，省钱！ ↓ Ingress Controller (Nginx / Traefik) ↓ Ingress (路由规则) ├── /api/* → api-service (端口 8080) └── /web/* → web-service (端口 80)

⚖️ 架构取舍：Ingress Controller 选择

✅ Nginx Ingress (最流行)
  • 社区成熟，文档丰富
  • 性能稳定
✅ Traefik (现代化)
  • 自动服务发现
  • 内置 Let's Encrypt 支持

实践建议：新手用 Nginx (资料多)，高级场景用 Traefik (功能强)。

Ingress - Layer-7-Routing (Ingress - L7 Routing)

Warum benötigt: Service (LoadBalancer) benötigt einen LB pro Service, zu teuer! Ingress bietet HTTP/HTTPS-Routing.

🏗️ Ingress-Architektur (Ingress Architecture)

Internet ↓ LoadBalancer (AWS ELB) - 1 Stück, spart Geld! ↓ Ingress Controller (Nginx / Traefik) ↓ Ingress (Routing-Regeln) ├── /api/* → api-service (Port 8080) └── /web/* → web-service (Port 80)

⚖️ Architektur-Trade-off: Ingress Controller-Auswahl

✅ Nginx Ingress (beliebteste)
  • Ausgereifte Community, umfangreiche Dokumentation
  • Stabile Leistung
✅ Traefik (modern)
  • Automatische Service-Discovery
  • Integriertes Let's Encrypt Support

Praxis-Empfehlung: Anfänger: Nginx (mehr Materialien), Fortgeschrittene: Traefik (mehr Funktionen).

安全最佳实践 (Security Best Practices)

核心原则：最小权限、纵深防御、持续监控。

关键安全措施：

  • RBAC：基于角色的权限控制
  • Network Policy：网络隔离 (类似防火墙规则)
  • Pod Security Standards：限制 Pod 特权
  • Image Scanning：扫描镜像漏洞
  • Secrets Encryption：加密 etcd 中的 Secret
💡 实际例子： Network Policy 限制访问 
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: db-network-policy
spec:
  podSelector:
    matchLabels:
      app: mysql
  policyTypes:
  - Ingress

⚖️ 架构取舍：安全 vs 易用性

✅ 严格安全 (生产必须)
  • 启用 RBAC + Network Policy
  • 镜像签名验证
  • 定期扫描漏洞
❌ 宽松配置 (开发可用)
  • 禁用 Network Policy
  • 使用 latest 标签
  • 开发速度快，但风险高

Sicherheits-Best-Practices (Security Best Practices)

Kernprinzipien: Geringste Rechte, Verteidigung in der Tiefe, kontinuierliche Überwachung.

Wichtige Sicherheitsmaßnahmen:

  • RBAC: Rollenbasierte Zugriffskontrolle
  • Network Policy: Netzwerkisolation (ähnlich Firewall-Regeln)
  • Pod Security Standards: Pod-Privilegien einschränken
  • Image Scanning: Image-Schwachstellen scannen
  • Secrets Encryption: Secrets in etcd verschlüsseln
💡 Praxis-Beispiel: Network Policy Zugriff einschränken 
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: db-network-policy
spec:
  podSelector:
    matchLabels:
      app: mysql
  policyTypes:
  - Ingress

⚖️ Architektur-Trade-off: Sicherheit vs Benutzerfreundlichkeit

✅ Strenge Sicherheit (Produktion erforderlich)
  • RBAC + Network Policy aktivieren
  • Image-Signatur-Verifizierung
  • Regelmäßige Schwachstellen-Scans
❌ Lockerere Konfiguration (Entwicklung möglich)
  • Network Policy deaktivieren
  • latest-Tags verwenden
  • Schnellere Entwicklung, aber höheres Risiko

EKS 集群搭建 (EKS Cluster Setup)

为什么选择 EKS：AWS 托管的控制平面，高可用且无运维负担。

🏗️ EKS 架构 (EKS Architecture)

AWS EKS Cluster ├── Control Plane (AWS 托管) │ ├── API Server (高可用) │ ├── etcd (多可用区备份) │ └── IAM Role └── Worker Nodes ├── Managed Node Groups (推荐) └── Fargate (Serverless)
💡 实际例子： 使用 eksctl 创建 EKS 集群 
eksctl create cluster \
  --name my-cluster \
  --version 1.28 \
  --region us-east-1 \
  --nodegroup-name workers \
  --node-type t3.medium \
  --nodes 3

⚖️ 架构取舍：EKS Node 类型选择

✅ EC2 (Managed Node Groups)
  • 成本低 (长期运行)
  • 完全控制
  • 适合稳定负载
✅ Fargate (Serverless)
  • 无需管理 Node
  • 按 Pod 计费
  • 适合波动负载

🏗️ EKS-Architektur (EKS Architecture)

AWS EKS Cluster ├── Control Plane (AWS verwaltet) │ ├── API Server (hochverfügbar) │ ├── etcd (Multi-AZ-Backup) │ └── IAM Role └── Worker Nodes ├── Managed Node Groups (empfohlen) └── Fargate (Serverless)

核心目标：系统在任何单点故障下仍然可用 (99.99% SLA)。

🏗️ 高可用架构 (HA Architecture)

Multi-AZ Deployment ├── AZ-1a → Worker Nodes (2 台) ├── AZ-1b → Worker Nodes (2 台) └── AZ-1c → Worker Nodes (2 台) + Pod Anti-Affinity + PodDisruptionBudget (PDB) + Horizontal Pod Autoscaler (HPA)

        values:
        - nginx
    topologyKey: kubernetes.io/hostname

核心策略：
