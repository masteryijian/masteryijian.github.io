基础准备阶段

Linux 基础 (Linux Fundamentals)

为什么重要：K8s 底层运行在 Linux 上，需要熟悉命令行、进程管理、网络基础。

核心知识点：

  • 文件系统结构 (Filesystem Hierarchy)
  • 进程管理 (Process Management) - ps, top, systemd
  • 网络基础 (Networking Basics) - IP, DNS, iptables
  • 权限管理 (Permission Management) - user, group, chmod

📝 练习任务

在本地虚拟机或云服务器上：

  • 熟练使用 vim / nano 编辑器
  • 配置 SSH 免密登录
  • 设置防火墙规则 (iptables / firewalld)
  • 查看系统资源使用情况 (df, free, top)

Docker 基础 (Docker Fundamentals)

核心概念：容器化技术的基石，K8s 的底层运行时。

🏗️ Docker 架构 (Docker Architecture)

Client (Docker CLI) ↓ Docker Daemon (dockerd) ↓ Container Runtime (containerd / runc) ↓ Containers (隔离的进程空间)

必须掌握：

  • 镜像 (Image)：只读模板，分层存储 (Layer Storage)
  • 容器 (Container)：镜像的运行实例，轻量级虚拟化
  • Dockerfile：构建镜像的指令文件
  • Registry：镜像存储仓库 (如 Docker Hub)
💡 实际例子： 构建一个 Node.js 应用镜像 
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install --production
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]

⚖️ 架构取舍：镜像大小 vs 构建速度

场景：多阶段构建 (Multi-stage Build) 的选择

✅ 多阶段构建
  • 最终镜像体积小 (去掉构建工具)
  • 安全性更高 (减少攻击面)
  • 适合生产环境
❌ 单阶段构建
  • 构建速度快 (无需多步)
  • 调试方便 (包含完整工具)
  • 适合开发环境

实践建议：开发用单阶段（快速迭代），生产用多阶段（优化体积）。

📝 练习任务

  • 安装 Docker Desktop (Mac/Windows) 或 Docker Engine (Linux)
  • 拉取并运行 Nginx 镜像：docker run -d -p 80:80 nginx
  • 编写 Dockerfile 构建自定义镜像
  • 使用 Docker Compose 编排多容器应用
  • 推送镜像到 Docker Hub 或私有 Registry

AWS 云平台基础

AWS 核心服务 (AWS Core Services)

为什么重要：AWS 是 K8s 最常见的运行环境 (EKS)，需要理解云基础设施。

🏗️ AWS VPC 网络架构 (VPC Networking Architecture)

VPC (Virtual Private Cloud) - 10.0.0.0/16 ├── Public Subnet (10.0.1.0/24) │ ├── Internet Gateway (IGW) │ └── NAT Gateway ├── Private Subnet (10.0.2.0/24) │ └── EC2 Instances / EKS Worker Nodes └── Private Subnet (10.0.3.0/24) └── Database (RDS / Aurora)

核心服务：

  • EC2 (Elastic Compute Cloud)：虚拟服务器，K8s 节点的运行载体
  • VPC (Virtual Private Cloud)：隔离的虚拟网络环境
  • EBS (Elastic Block Store)：持久化块存储，类似硬盘
  • ELB (Elastic Load Balancing)：负载均衡器，流量入口
  • IAM (Identity and Access Management)：权限管理，安全控制
  • S3 (Simple Storage Service)：对象存储，存镜像/日志/备份
💡 实际例子： 在 AWS 上手动部署 K8s 前的网络规划
  • 创建 VPC：10.0.0.0/16 (65536 个 IP)
  • 公有子网：10.0.1.0/24 (256 个 IP，放 ELB)
  • 私有子网：10.0.2.0/24 (256 个 IP，放 Worker Nodes)
  • 配置 NAT Gateway：让私有子网访问互联网 (下载镜像)

⚖️ 架构取舍：托管 K8s (EKS) vs 自管 K8s (kOps / kubeadm)

✅ AWS EKS (Managed)
  • 控制平面高可用 (AWS 负责)
  • 自动升级和补丁
  • 集成 AWS 服务 (IAM, ELB, EBS)
  • 按小时计费 (无控制平面预付)
❌ 自管 K8s (Self-managed)
  • 完全控制 (自定义网络/CNI)
  • 无厂商锁定 (Multi-cloud)
  • 学习价值高 (理解底层)
  • 成本低 (仅 EC2 费用)

实践建议：生产用 EKS (省心)，学习用 kubeadm (深入)。本书两个都会讲。

📝 练习任务

  • 注册 AWS 账号 (需要信用卡，有免费套餐)
  • 启动一个 EC2 实例 (t2.micro，免费套餐)
  • 创建 VPC 和子网，配置安全组 (Security Group)
  • 挂载 EBS 卷到 EC2，测试持久化存储
  • 配置 IAM 用户和角色，理解最小权限原则

Kubernetes 核心概念

K8s 架构 (Kubernetes Architecture)

🏗️ K8s 集群架构 (Cluster Architecture)

Control Plane (Master Node) - 大脑 ├── API Server (kube-apiserver) - 统一入口 ├── etcd - 分布式键值存储 (集群状态) ├── Scheduler (kube-scheduler) - 调度 Pod 到 Node ├── Controller Manager (kube-controller-manager) - 维护期望状态 └── Cloud Controller Manager - 对接云平台 (可选) Worker Nodes - 干活的手 ├── Kubelet - 与 API Server 通信，管理 Pod ├── Kube-proxy - 网络代理，Service 实现 ├── Container Runtime (containerd / Docker) - 运行容器 └── Pods - 最小调度单元 (包含 1+ 容器)

核心原理：

  • 声明式 API (Declarative API)：告诉 K8s "想要什么状态"，而不是"怎么做"
  • 控制循环 (Control Loop)：不断对比 "当前状态" 和 "期望状态"，自动修复
  • 松耦合 (Loose Coupling)：各组件通过 API Server 通信，互不依赖
💡 实际例子： 副本自愈 (Self-healing)

你声明：replicas: 3 (想要 3 个 Pod)

某个 Pod 崩溃 → K8s 检测到 "当前 2 个" ≠ "期望 3 个" → 自动创建新 Pod

对比传统：需要手动 restart，或写 Supervisor 脚本。

Pod - 最小调度单元 (Pod - Smallest Deployable Unit)

为什么需要 Pod：容器共享网络/存储，类似 "豌豆荚" 的概念。

🏗️ Pod 内部结构 (Pod Internal Structure)

Pod (IP: 10.244.1.5) ├── Pause Container (基础设施容器，持有网络命名空间) ├── App Container (你的应用，如 Nginx) └── Sidecar Container (辅助容器，如日志收集)

关键特性：

  • 共享网络 (Shared Network)：Pod 内所有容器共享 IP 和端口空间
  • 共享存储 (Shared Storage)：Volume 可以被多个容器挂载
  • 生命周期 (Lifecycle)：Pod 是短暂的，随时可能被重建

⚖️ 架构取舍：一个 Pod 运行多个容器 vs 每个 Pod 一个容器

✅ Sidecar 模式 (多容器 Pod)
  • 紧密耦合的服务 (如日志收集)
  • 共享存储和网络 (高效通信)
  • 统一生命周期 (一起扩缩容)
❌ 单容器 Pod
  • 独立扩缩容 (精细化资源控制)
  • 故障隔离 (一个挂了不影响其他)
  • 符合微服务原则 (单一职责)

实践建议：除非是 Sidecar (如 Istio Proxy)，否则一个 Pod 只放一个容器。

📝 练习任务

  • 安装 Minikube (本地 K8s 单机版)
  • 运行第一个 Pod：kubectl run nginx --image=nginx
  • 查看 Pod 详情：kubectl describe pod nginx
  • 进入 Pod 内部：kubectl exec -it nginx -- /bin/bash
  • 删除 Pod 观察重建 (配合 Deployment)

工作负载控制器 (Workload Controllers)

为什么需要控制器：直接管理 Pod 太累，控制器提供声明式管理和自愈能力。

核心控制器对比：

  • Deployment：无状态应用 (Stateless)，支持滚动更新 (Rolling Update)
  • StatefulSet：有状态应用 (Stateful)，稳定的网络标识和存储
  • DaemonSet：每个 Node 运行一个副本 (如日志收集器)
  • Job / CronJob：批处理任务 (Batch Jobs)
💡 实际例子： Deployment 的滚动更新策略 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 最多多 1 个 Pod
      maxUnavailable: 1   # 最多少 1 个 Pod
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        ports:
        - containerPort: 80

效果：更新镜像时，先启动新 Pod，再删除旧 Pod，保证服务不中断。

⚖️ 架构取舍：Deployment vs StatefulSet

✅ Deployment (无状态)
  • Pod 名称随机 (nginx-xxxx-xxx)
  • 扩缩容快 (无需顺序)
  • 适合 Web 服务、API
✅ StatefulSet (有状态)
  • Pod 名称固定 (mysql-0, mysql-1)
  • 稳定的网络标识 (Persistent Identity)
  • 适合数据库、消息队列

实践建议：99% 的场景用 Deployment，只有需要稳定标识时才用 StatefulSet。

网络与存储

K8s 网络模型 (Kubernetes Networking Model)

核心原则：所有 Pod 可以互相通信，所有 Pod 可以与所有 Node 通信。

🏗️ K8s 网络架构 (Network Architecture)

Service (虚拟 IP，负载均衡) ↓ Pod (10.244.1.5) ←→ Pod (10.244.2.3) [Pod 间通信] ↓ Node (物理机) ↓ Internet / External Service

三层网络模型：

  • Container-to-Container (同一 Pod)：localhost 通信
  • Pod-to-Pod (跨 Node)：Overlay 网络 (如 Flannel, Calico)
  • Pod-to-Service：通过 kube-proxy 实现负载均衡
  • External-to-Service：NodePort / LoadBalancer / Ingress
💡 实际例子： Service 的负载均衡 
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: ClusterIP
  selector:
    app: nginx
  ports:
    - protocol: TCP
      port: 80
      targetPort: 80

效果：访问 nginx-service:80 会自动负载均衡到 3 个 Pod。

⚖️ 架构取舍：Service 类型选择 (ClusterIP vs NodePort vs LoadBalancer)

✅ ClusterIP (默认)
  • 仅在集群内可访问
  • 适合后端服务 (如数据库)
  • 安全 (不暴露到外网)
✅ LoadBalancer (生产推荐)
  • 自动创建云平台负载均衡器
  • 高可用 (跨可用区)
  • 成本高 (每个 Service 一个 LB)

实践建议：外部流量用 Ingress (七层) + LoadBalancer (四层)，内部通信用 ClusterIP。

持久化存储 (Persistent Storage)

问题：Pod 是短暂的，数据不能存在容器里 (重启就没了)。

核心概念：

  • PV (PersistentVolume)：集群级别的存储资源 (由管理员创建)
  • PVC (PersistentVolumeClaim)：用户对存储的请求 (类似 "下单")
  • StorageClass：动态供应 (Dynamic Provisioning)，自动创建 PV
💡 实际例子： AWS EBS 作为持久化存储 
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: gp3-storage
provisioner: kubernetes.io/aws-ebs
parameters:
  type: gp3
  fsType: ext4
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mysql-pvc
spec:
  accessModes:
    - ReadWriteOnce  # 只能被一个 Node 挂载
  storageClassName: gp3-storage
  resources:
    requests:
      storage: 20Gi

⚖️ 架构取舍：存储访问模式 (Access Modes)

✅ ReadWriteOnce (RWO)
  • 只能被一个 Node 挂载
  • 性能最好 (块存储，如 EBS)
  • 适合数据库
✅ ReadWriteMany (RWX)
  • 可以被多个 Node 同时挂载
  • 适合共享文件系统 (如 NFS, EFS)
  • 性能较差 (网络文件系统)

实践建议：数据库用 RWO (EBS)，共享配置用 RWX (EFS/NFS)。

配置管理与安全

ConfigMap 与 Secret (Configuration & Secrets Management)

为什么需要：将配置与代码分离 (12-Factor App 原则)。

核心区别：

  • ConfigMap：存储非敏感配置 (如配置文件、环境变量)
  • Secret：存储敏感信息 (如密码、API Key)，base64 编码 (不是加密！)
💡 实际例子： 使用 ConfigMap 注入配置 
# 创建 ConfigMap
kubectl create configmap app-config --from-file=app.conf

# 在 Pod 中使用
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  containers:
  - name: app
    image: myapp
    env:
    - name: DATABASE_URL
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: database.url
    volumeMounts:
    - name: config-volume
      mountPath: /etc/config
  volumes:
  - name: config-volume
    configMap:
      name: app-config

⚖️ 架构取舍：配置注入方式 (环境变量 vs 文件挂载)

✅ 环境变量 (env)
  • 简单直接
  • 适合简单配置 (如 API_URL)
  • 应用无需修改代码
✅ 文件挂载 (volumeMount)
  • 支持复杂配置 (如 Nginx.conf)
  • 支持动态更新 (修改 ConfigMap 后自动同步)
  • 适合配置文件

实践建议：简单 K/V 用环境变量，配置文件用挂载。

📝 练习任务

  • 创建 ConfigMap 和 Secret
  • 在 Pod 中通过环境变量和文件挂载使用配置
  • 使用 kubectl edit 修改 ConfigMap，观察应用是否自动更新
  • 集成 AWS Secrets Manager (使用 External Secrets Operator)

高级主题与最佳实践

Helm - K8s 包管理器 (Helm - Package Manager)

为什么需要：K8s YAML 太多太复杂，Helm 提供模板化和版本管理。

核心概念：

  • Chart：Helm 的包格式 (类似 Debian 的 .deb)
  • Release：Chart 的安装实例 (可以装多个版本)
  • Repository：Chart 的仓库 (类似 Docker Hub)
  • Values：自定义配置 (覆盖默认值)
💡 实际例子： 使用 Helm 安装 Nginx Ingress Controller 
# 添加仓库
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo update

# 安装 (创建 Release)
helm install my-ingress ingress-nginx/ingress-nginx

# 自定义配置 (通过 values.yaml)
helm install my-ingress ingress-nginx/ingress-nginx \
  --set controller.replicaCount=2 \
  --set controller.service.type=LoadBalancer

📝 练习任务

  • 安装 Helm (v3+)
  • 搜索并安装 Chart：helm search repo mysql
  • 创建自己的 Helm Chart：helm create myapp
  • 使用 helm upgrade 和 helm rollback 管理版本

Ingress - 七层路由 (Ingress - L7 Routing)

为什么需要：Service (LoadBalancer) 每个服务需要一个 LB，太贵！Ingress 提供 HTTP/HTTPS 路由。

🏗️ Ingress 架构 (Ingress Architecture)

Internet ↓ LoadBalancer (AWS ELB) - 1 个，省钱！ ↓ Ingress Controller (Nginx / Traefik / HAProxy) ↓ Ingress (路由规则) ├── /api/* → api-service (端口 8080) └── /web/* → web-service (端口 80)
💡 实际例子： Ingress 路由配置 
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  rules:
  - host: example.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 8080
      - path: /
        pathType: Prefix
        backend:
          service:
            name: web-service
            port:
              number: 80
  tls:
  - hosts:
    - example.com
    secretName: tls-secret  # HTTPS 证书

⚖️ 架构取舍：Ingress Controller 选择 (Nginx vs Traefik vs HAProxy)

✅ Nginx Ingress (最流行)
  • 社区成熟，文档丰富
  • 性能稳定
  • 适合大多数场景
✅ Traefik (现代化)
  • 自动服务发现 (与 K8s API 集成)
  • 内置 Let's Encrypt 支持 (自动 HTTPS)
  • Dashboard 可视化

实践建议：新手用 Nginx (资料多)，高级场景用 Traefik (功能强)。

监控与日志 (Monitoring & Logging)

为什么重要：分布式系统必须可观测 (Observability)，否则出问题不知道为啥。

🏗️ K8s 可观测性架构 (Observability Stack)

Monitoring (指标) ├── Prometheus - 采集指标 (Metrics) ├── Grafana - 可视化仪表盘 (Dashboards) └── AlertManager - 告警 (Alerting) Logging (日志) ├── EFK Stack (Elasticsearch + Fluentd + Kibana) └── Loki + Promtail + Grafana (轻量级) Tracing (链路追踪) └── Jaeger / Zipkin - 分布式追踪 (Distributed Tracing)
💡 实际例子： Prometheus 监控 K8s 集群
  • 安装 Prometheus：helm install prometheus prometheus-community/kube-prometheus-stack
  • 自动采集：Node Exporter (节点指标)、Kube-state-metrics (K8s 对象状态)
  • 查看仪表盘：kubectl port-forward svc/prometheus-grafana 3000:80
  • 设置告警规则：CPU 使用率 > 80% 持续 5 分钟 → 发送 Slack 通知

📝 练习任务

  • 安装 Prometheus + Grafana (用 Helm)
  • 导入社区 Dashboard (如 Node Exporter Full)
  • 配置告警规则 (AlertManager)
  • 集成 Loki 收集日志

安全最佳实践 (Security Best Practices)

核心原则：最小权限、纵深防御、持续监控。

关键安全措施：

  • RBAC (Role-Based Access Control)：基于角色的权限控制
  • Network Policy：网络隔离 (类似防火墙规则)
  • Pod Security Standards：限制 Pod 特权 (如禁止 root)
  • Image Scanning：扫描镜像漏洞 (如 Trivy)
  • Secrets Encryption：加密 etcd 中的 Secret (不是 base64！)
💡 实际例子： Network Policy 限制访问 
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: db-network-policy
spec:
  podSelector:
    matchLabels:
      app: mysql
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: api  # 只允许 app=api 的 Pod 访问
    ports:
    - protocol: TCP
      port: 3306

效果：只有标签为 app=api 的 Pod 可以访问 MySQL，其他都被拒绝 (默认拒绝策略)。

⚖️ 架构取舍：安全 vs 易用性

✅ 严格安全 (生产必须)
  • 启用 RBAC + Network Policy
  • 镜像签名验证 (Image Signing)
  • 定期扫描漏洞
  • 审计日志 (Audit Logs)
❌ 宽松配置 (开发可用)
  • 禁用 Network Policy (方便调试)
  • 使用 latest 标签 (快速迭代)
  • 共享 Docker Registry 凭证
  • 开发速度快，但风险高

实践建议：开发环境宽松，生产环境严格 (通过 CI/CD 自动化安全检查)。

AWS EKS 实战部署

EKS 集群搭建 (EKS Cluster Setup)

为什么选择 EKS：AWS 托管的控制平面，高可用且无运维负担。

🏗️ EKS 架构 (EKS Architecture)

AWS EKS Cluster ├── Control Plane (AWS 托管，用户不可见) │ ├── API Server (高可用) │ ├── etcd (多可用区备份) │ └── IAM Role (权限控制) └── Worker Nodes (用户管理) ├── Managed Node Groups (推荐) └── Fargate (Serverless，无需管理 Node)
💡 实际例子： 使用 eksctl 创建 EKS 集群 
# 安装 eksctl
brew install eksctl  # Mac
# 或
curl --silent --location "https://github.com/weaveworks/eksctl/releases/latest/download/eksctl_$(uname -s)_amd64.tar.gz" | tar xz -C /tmp
sudo mv /tmp/eksctl /usr/local/bin

# 创建集群 (自动创建 VPC、Node Group)
eksctl create cluster \
  --name my-cluster \
  --version 1.28 \
  --region us-east-1 \
  --nodegroup-name workers \
  --node-type t3.medium \
  --nodes 3 \
  --nodes-min 1 \
  --nodes-max 5 \
  --managed

# 配置 kubectl
aws eks update-kubeconfig --name my-cluster --region us-east-1
kubectl get nodes  # 验证连接

⚖️ 架构取舍：EKS Node 类型选择 (EC2 vs Fargate)

✅ EC2 (Managed Node Groups)
  • 成本低 (长期运行)
  • 完全控制 (自定义 AMI)
  • 适合稳定负载
✅ Fargate (Serverless)
  • 无需管理 Node (省心)
  • 按 Pod 计费 (适合波动负载)
  • 启动慢 (冷启动问题)

实践建议：稳定负载用 EC2 (省钱)，波动负载用 Fargate (灵活)。混合使用也可以。

  • 配置 IAM Roles for Service Accounts (IRSA) - 让 Pod 访问 AWS 服务
  • 部署应用到 EKS，暴露 LoadBalancer Service
  • 配置 ALB Ingress Controller (AWS Load Balancer Controller)
  • 使用 EBS/EFS CSI Driver 挂载持久化存储

CI/CD 集成 (CI/CD Integration)

为什么重要：手动部署太慢且容易出错，自动化是云原生的核心。

🏗️ K8s CI/CD 流程 (CI/CD Pipeline)

GitHub / GitLab ↓ (git push) CI Server (GitHub Actions / GitLab CI / Jenkins) ├── 1. 代码检查 (Lint) ├── 2. 单元测试 (Unit Test) ├── 3. 构建镜像 (Docker Build) ├── 4. 推送镜像 (Docker Push to ECR) └── 5. 部署到 K8s (kubectl apply / Helm upgrade) ↓ EKS Cluster └── New Version Rollout (滚动更新)
💡 实际例子： GitHub Actions 部署到 EKS 
# .github/workflows/deploy.yml
name: Deploy to EKS
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v3
    
    - name: Configure AWS Credentials
      uses: aws-actions/configure-aws-credentials@v2
      with:
        aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
        aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
        aws-region: us-east-1
    
    - name: Login to Amazon ECR
      id: login-ecr
      uses: aws-actions/amazon-ecr-login@v1
    
    - name: Build and Push Image
      env:
        ECR_REGISTRY: ${{ steps.login-ecr.outputs.registry }}
        IMAGE_TAG: ${{ github.sha }}
      run: |
        docker build -t $ECR_REGISTRY/myapp:$IMAGE_TAG .
        docker push $ECR_REGISTRY/myapp:$IMAGE_TAG
    
    - name: Update Kubeconfig
      run: aws eks update-kubeconfig --name my-cluster --region us-east-1
    
    - name: Deploy to EKS
      run: |
        kubectl set image deployment/myapp myapp=$ECR_REGISTRY/myapp:$IMAGE_TAG
        kubectl rollout status deployment/myapp

生产环境最佳实践

高可用性设计 (High Availability Design)

核心目标：系统在任何单点故障下仍然可用 (99.99% SLA)。

🏗️ 高可用架构 (HA Architecture)

Multi-AZ Deployment (多可用区部署) ├── AZ-1a (可用区 1) │ └── Worker Nodes (2 台) ├── AZ-1b (可用区 2) │ └── Worker Nodes (2 台) └── AZ-1c (可用区 3) └── Worker Nodes (2 台) + Pod 反亲和性 (Pod Anti-Affinity) - 确保副本分散在不同 Node + PodDisruptionBudget (PDB) - 确保至少 N 个 Pod 始终运行 + Horizontal Pod Autoscaler (HPA) - 自动扩缩容
💡 实际例子： Pod Anti-Affinity 配置 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 3
  template:
    spec:
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: app
                operator: In
                values:
                - nginx
            topologyKey: kubernetes.io/hostname  # 不同 Node
      containers:
      - name: nginx
        image: nginx

效果：3 个副本必须运行在不同 Node 上，避免单点故障。

成本优化 (Cost Optimization)

为什么重要：云资源很贵，优化成本 = 提高利润。

核心策略：

  • Right Sizing：选择合适大小的实例 (不要盲目选大的)
  • Spot Instances：使用竞价实例 (便宜 70-90%，但可能被回收)
  • Auto Scaling：按需自动扩缩容 (高峰期扩容，低峰期缩容)
  • Namespace Quotas：限制团队资源使用 (防止滥用)
💡 实际例子： 使用 Spot Instance 节省成本 
# eksctl 创建混合 Node Group (On-Demand + Spot)
eksctl create nodegroup \
  --cluster my-cluster \
  --name spot-workers \
  --node-type t3.medium \
  --nodes 3 \
  --nodes-min 0 \
  --nodes-max 10 \
  --spot \
  --instance-prefix spot

效果：Spot 实例便宜 70%，配合 Cluster Autoscaler 自动补充节点。

⚖️ 架构取舍：成本 vs 可靠性

✅ 省钱方案 (开发/测试)
  • 使用 Spot Instance
  • 单可用区部署
  • 关闭非高峰时段资源
✅ 高可靠方案 (生产)
  • On-Demand Instance (稳定)
  • 多可用区部署
  • 预留实例 (RI) 或 Savings Plans

实践建议：开发用 Spot，生产用 On-Demand + 多 AZ。

灾难恢复 (Disaster Recovery)

核心原则：备份 etcd，备份应用数据，定期演练恢复流程。

📝 灾难恢复清单

  • etcd 备份：每天自动备份 etcd 快照到 S3
  • 应用数据备份：数据库使用 AWS RDS (自动备份)，或 Velero (K8s 资源备份)
  • 跨区域复制：关键数据复制到另一个 AWS 区域 (如 us-east-1 → us-west-2)
  • 恢复演练：每季度演练一次恢复流程 (不能只备份不演练！)
💡 实际例子： 使用 Velero 备份 K8s 资源 
# 安装 Velero
velero install \
  --provider aws \
  --plugins velero/velero-plugin-for-aws:v1.6.0 \
  --bucket my-cluster-backups \
  --backup-location-config region=us-east-1 \
  --snapshot-location-config region=us-east-1

# 创建备份
velero backup create my-backup --include-namespaces default

# 恢复备份
velero restore create --from-backup my-backup
